# 智能对话式任务管理助手技术方案

## 1. 项目目标

基于 `nanobot` 现有项目中已经验证过的 Agent 抽象思路，开发一个面向面试作业的轻量化产品：

- 运行在 Cloudflare Worker 边缘环境
- 提供公开可访问的网页对话界面
- 支持自然语言任务管理
- 支持联网搜索与深度研究
- 支持文件上传、解析、向量化与 RAG 问答
- 支持用户信息、任务、对话、文件元数据的持久化

本方案的核心原则是：

- 借用 `nanobot` 的抽象，不直接搬运其完整运行时
- 优先保证功能闭环、代码清晰、部署简单
- 以面试交付为目标，尽可能轻量

## 2. 基于 nanobot 的改造思路

### 2.1 可复用的思想

`nanobot` 当前项目中有几类能力非常适合直接借鉴：

- LLM Provider 抽象层
- Agent Loop 的职责拆分方式
- Tool Call 驱动的交互模式
- 子代理规划的任务拆解思路
- 文件解析与搜索工具的模块化设计

这些能力在现有代码中已经有清晰体现：

- `nanobot/nanobot.py`
- `nanobot/providers/base.py`
- `nanobot/agent/loop.py`
- `nanobot/agent/subagent.py`
- `nanobot/agent/tools/web.py`
- `nanobot/utils/document.py`

### 2.2 不建议直接复用的部分

`nanobot` 当前更偏向“通用长运行 Agent 框架”，而本题是“单场景 Web 产品”。因此以下部分不建议直接沿用：

- `channels/` 多渠道接入体系
- `cli/` 命令行交互体系
- `cron/` 和 Dream 异步记忆系统
- 基于本地文件系统的 `workspace/memory` 设计
- `aiohttp` API Server 形态
- 执行本地 shell、读写本地工作区等工具

原因很直接：

- Cloudflare Worker 是无状态边缘运行时
- 面试作业不需要通用平台能力
- 过多保留原架构会显著增加实现复杂度和部署风险

结论是：

**保留抽象层，重写运行时外壳。**

## 3. 总体架构

### 3.1 技术选型

- 后端运行时：Cloudflare Python Worker
- Web 框架：轻量路由，必要时使用 FastAPI for Workers
- 前端：静态单页应用，优先 `HTML + Alpine.js + TailwindCSS`
- 关系数据库：Cloudflare D1
- 文件存储：Cloudflare R2
- 向量数据库：Qdrant Cloud
- 搜索服务：Serper.dev
- 大模型：Gemini 2.0 Flash Lite
- 向量化：单独抽象 Embedding Provider，先接低成本 Embedding API

### 3.2 架构分层

```text
Browser UI
  -> Cloudflare Worker API
    -> Agent Service
      -> Intent Router / Tool Dispatcher
      -> Planner / Research Orchestrator
      -> RAG Service
    -> D1
    -> R2
    -> Qdrant
    -> Gemini / Embedding API / Serper API
```

## 4. 核心功能设计

### 4.1 基础对话与用户管理

首次访问时，前端生成 `client_id` 并写入浏览器 Cookie 或 Local Storage。后端收到请求后：

- 若没有对应用户，创建匿名用户记录
- 若用户缺少姓名或邮箱，在系统提示词中注入“先补全用户资料”的规则
- 当用户提供姓名、邮箱后，写入 D1
- 后续对话从 D1 读取用户资料，确保 AI 正确称呼用户
- AI 昵称单独保存在 `assistant_settings` 表中，支持自然语言修改

实现上不使用传统表单，仍然保持对话式交互。也就是说，“请告诉我你的名字和邮箱”本身就是系统对话流程的一部分。

### 4.2 任务管理

任务管理采用 “自然语言 -> 意图识别 -> 工具调用 -> D1 CRUD” 模式。

系统为 LLM 暴露以下工具：

- `create_task`
- `update_task`
- `list_tasks`
- `delete_task`
- `get_task`

任务表结构建议如下：

- `id`
- `user_id`
- `title`
- `details`
- `status`
- `priority`
- `due_at`
- `created_at`
- `updated_at`

典型示例：

- “帮我创建一个下周五前完成的简历优化任务”
- “把刚才那个任务改成高优先级”
- “列出我所有未完成任务”
- “删除预算调研那个任务”

LLM 不直接生成 SQL，而是输出标准化工具调用参数，由后端服务层完成数据库操作。

### 4.3 外部搜索与深度研究

#### 普通搜索

当用户只是提问实时信息时，调用 `search_web(query)` 即可，返回搜索结果摘要，并在需要时抓取 1 到 3 个页面做补充整理。

#### 深度研究

当用户的问题具备研究特征时，例如：

- “帮我调研 AI Agent 面试作业常见实现方案”
- “对比 Cloudflare Worker 上做 RAG 的几种轻量架构”

系统切换到 Research Mode。

Research Mode 的流程为：

1. Planner 先将用户目标拆解为 3 到 5 个子问题
2. 每个子问题作为一个“逻辑子代理”执行 `搜索 -> 阅读 -> 摘要`
3. 主代理将所有子结果整合为结构化 Markdown 报告

这里的“子代理”不一定要实现成真正并发后台任务。面试作业中完全可以做成一次请求内的顺序规划执行，但在文档中明确体现：

- 有规划器
- 有子任务拆解
- 有中间结果整合

这已经足以体现 Agent 设计能力。

### 4.4 文件处理与 RAG

#### 文件存储设计

- 原始文件存入 R2
- 文件元数据存入 D1
- 文本块及向量存入 Qdrant

#### 文件处理流程

1. 用户上传文件
2. Worker 校验文件类型和大小
3. 文件写入 R2
4. 服务层解析文本内容
5. 文本按块切分
6. 调用 Embedding Provider 生成向量
7. 向量和 metadata 写入 Qdrant

#### 支持文件类型

优先实现：

- `.txt`
- `.md`
- `.pdf`
- `.docx`

如果考虑 Worker 运行时和包体积限制，建议采用如下轻量策略：

- `txt/md` 在 Worker 内直接处理
- `pdf/docx` 优先使用兼容 Python Worker 的轻量解析方案
- 若某些重依赖在 Worker 中不稳定，可以退而采用“前端抽文本、后端负责向量化”的折中方案

#### RAG 检索流程

当用户提问涉及上传文件时：

1. 先基于用户问题生成检索向量
2. 在 Qdrant 中按 `user_id` 和 `file_id` 过滤检索
3. 返回 Top-K 相关片段
4. 将片段拼装进上下文
5. 交给主模型生成回答

#### 文件管理能力

前端工作区应提供最小但完整的文件管理能力：

- 文件列表
- 文件类型标签
- 上传进度状态
- 删除文件
- 可选的重命名

### 4.5 数据持久化

#### D1 负责

- 用户信息
- AI 昵称设置
- 对话会话
- 对话消息
- 任务列表
- 文件元数据

#### Qdrant 负责

- 文档分块向量
- 长期对话记忆摘要向量

#### R2 负责

- 文件原始二进制对象

这种职责划分的优点是：

- D1 结构化查询简单
- Qdrant 语义检索专职
- R2 适合文件原件管理
- 整体架构清楚，答辩时容易说明

## 5. Agent 设计

### 5.1 Agent Loop

主 Agent 的处理流程如下：

1. 读取用户信息与 AI 昵称
2. 读取最近对话
3. 判断是否需要补全用户资料
4. 判断用户意图
5. 若是任务类请求，调用任务工具
6. 若是搜索类请求，调用搜索工具
7. 若是研究类请求，进入 Planner
8. 若是知识问答类请求，触发 RAG 检索
9. 汇总工具结果并生成最终回复
10. 持久化消息记录与必要记忆

### 5.2 意图识别

为了控制复杂度，不建议实现一个重型分类器。可以让模型输出固定枚举：

- `collect_user_profile`
- `task_crud`
- `search_web`
- `deep_research`
- `file_qa`
- `general_chat`

这样实现简单，同时可以支持动态 Prompt 模板切换，满足加分项里的“细粒度意图识别”要求。

### 5.3 子代理规划

本项目中的子代理规划采用“逻辑子代理”模式：

- Planner 负责拆解问题
- 每个子问题进入统一的 research worker
- worker 负责搜索、抓取和压缩
- 主代理负责汇总

示例：

用户问题：  
“帮我研究一下面试作业里为什么推荐用 Python Worker + D1 + Qdrant，而不是把 nanobot 全量迁移过去。”

可拆成以下子问题：

- Cloudflare Python Worker 当前能力边界
- `nanobot` 当前架构中哪些模块不适合 Worker
- 轻量化替代方案如何满足题目要求
- 面试交付时如何平衡完整度与工程复杂度

最后汇总成一份带结论、优缺点、建议的报告。

### 5.4 记忆召回设计

采用双层记忆：

- 短期记忆：D1 中最近若干轮消息
- 长期记忆：Qdrant 中抽取后的用户偏好、项目事实、历史结论

回答时按以下顺序构造上下文：

1. 系统角色设定
2. 用户档案
3. 最近对话
4. 相关任务
5. 文件检索片段
6. 长期记忆摘要
7. 当前问题

相比 `nanobot` 当前基于本地文件的记忆方式，这一版更适合云端多用户 Web 产品。

## 8. 前端设计

前端不追求重型 SPA，重点是清晰、好看、够用。

页面建议分为左右两栏：

- 左侧：会话区
- 右侧：任务与文件工作区

界面应包含：

- 对话消息列表
- 输入框与发送按钮
- 文件上传按钮
- 文件列表
- 任务概览卡片
- 研究模式结果展示区

前端体验上的加分点：

- 上传进度提示
- 文件类型标签
- 对话中的结构化卡片
- 研究报告支持 Markdown 渲染

## 9. 轻量化实现策略

为了避免面试作业失控，建议主动收敛：

- 不做多渠道接入
- 不做真正分布式并发子代理
- 不做复杂权限体系
- 不做重型前端框架
- 不做过多模型适配器

优先保留以下“足够体现能力”的点：

- Tool Call
- Planner
- RAG
- D1 + Qdrant + R2
- 可部署
- 文档清楚

## 10. 风险与解决方案

### 风险 1：Python Worker 生态仍不如传统 Python 服务成熟

解决方案：

- 控制依赖数量
- 尽量使用纯 Python 或远程 API
- 将复杂解析逻辑封装为可替换服务

### 风险 2：文件解析在 Worker 中可能受限

解决方案：

- 优先支持文本类文件
- PDF/DOCX 采用最小必要依赖
- 必要时采用“前端抽文本 + 后端向量化”的降级策略

### 风险 3：深度研究链路太长，响应时间偏高

解决方案：

- 限制子问题数量
- 限制每个子问题抓取的网页数
- 研究模式单独展示“处理中”状态

### 风险 4：Qdrant 与 Embedding API 成本和配额

解决方案：

- 控制 chunk 数量和最大文件大小
- 只为有效文本生成向量
- 删除文件时同步清理向量

## 11. 面试交付建议

最终建议提交内容包括：

- 可运行源码
- `workers.dev` 公网地址
- `README.md`
- 本文档
- 一个简洁的架构图

演示顺序建议：

1. 首次对话收集用户资料
2. 自然语言创建与修改任务
3. 联网搜索一个实时问题
4. 发起一个研究任务并生成报告
5. 上传文件并进行 RAG 问答
6. 展示文件管理界面

## 12. 最终结论

本项目最优策略不是把 `nanobot` 现有框架整体迁移到 Cloudflare Worker，而是：

- 复用其 Agent 抽象思想
- 保留 Tool Call、Planner、Memory、RAG 的核心设计
- 重写适合 Worker 的轻量化运行时

这样既能满足题目要求，也能在面试中清楚展示：

- 你理解 Agent 的核心机制
- 你能做工程取舍
- 你能基于现有框架进行合理裁剪和重构

这比“功能很多但系统很重”更适合作为面试作业提交。
