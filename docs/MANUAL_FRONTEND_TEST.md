# 手动前端测试说明

本文档面向面试演示与作业验收，目标是让面试官可以从真实网页入口出发，按用户视角验证系统是否完成了以下闭环：

- 对话式用户建档
- 对话式任务管理
- 联网搜索与深度研究
- 文件上传、文件管理与 RAG 问答
- 会话与持久化数据的区分

建议结合公开演示地址一起使用：

- [TaskMate Worker Demo](https://taskmate-homework.keximing-taskmate.workers.dev/)

## 一、测试前准备

建议提前准备以下测试文件，方便现场演示。

### 1. `resume.md`

```md
# 项目经历
- 负责 Agent 平台设计
- 落地 RAG、文档解析、向量检索
- 使用 Cloudflare Worker + D1 + Qdrant 交付演示系统
```

### 2. `notes.txt`

```txt
面试准备任务：
1. 完成作业演示
2. 准备 Cloudflare Worker 架构说明
3. 准备 RAG 检索流程讲解
```

### 3. 可选图片文件

如果已经启用 OCR 多模态能力，可以额外准备一张带中文文字的图片，用来测试图片内容提取与问答。

## 二、页面主要区域说明

打开系统首页后，建议先给面试官快速说明页面结构：

- 左侧是任务列表与文件工作区
- 中间是聊天主面板
- 下方输入框用于自然语言交互
- 页面中部有研究任务进度卡片
- 顶部按钮支持新会话、任务刷新、复制/导出研究报告、清空数据

## 三、手动测试步骤

以下步骤按推荐演示顺序编排，适合现场直接走一遍。

### 1. 首次进入与用户建档

**操作步骤**

1. 打开首页。
2. 不提供姓名和邮箱，直接输入：`帮我创建一个任务`
3. 观察 AI 回复。
4. 再输入：`我叫小李，我的邮箱是 xiaoli@example.com`

**预期结果**

- 系统不会直接执行任务，而是先追问用户姓名和邮箱。
- 补全信息后，AI 会明确表示已经记住用户信息。
- 后续对话中，系统能持续使用该身份上下文。

### 2. 修改 AI 昵称

**操作步骤**

1. 在聊天框输入：`以后叫你阿塔`
2. 发送后观察页面左上角品牌名和聊天中的助手称呼。
3. 刷新页面，确认是否保留。

**预期结果**

- 助手昵称会被修改为“阿塔”。
- 页面中的品牌名与助手称呼同步变化。
- 刷新后昵称仍然存在，说明已持久化。

### 3. 创建任务

**操作步骤**

1. 输入：`帮我创建一个“面试作业”任务，要求突出 Agent、RAG、Cloudflare Worker 项目经验，下周五前完成，高优先级`
2. 发送后查看聊天区回复。
3. 观察左侧任务列表区域。

**预期结果**

- 聊天区提示任务已创建。
- 左侧任务列表出现该任务。
- 任务卡片中能看到标题、优先级、截止时间、更新时间。
- 如果任务包含需求描述，需求会一并展示。

### 4. 查询任务

**操作步骤**

1. 输入：`列出我的任务`
2. 或点击左侧 `刷新` 按钮。

**预期结果**

- 当前用户的任务列表能正确展示。
- 刚创建的“面试作业”任务应可见。

### 5. 更新任务

**操作步骤**

1. 输入：`把“面试作业”改成中优先级`
2. 再输入：`给这个任务补充要求：增加子代理规划、记忆召回、文件处理流程说明`
3. 再次输入：`列出我的任务`

**预期结果**

- 任务优先级从高优先级变为中优先级。
- 任务需求描述被更新。
- 任务列表展示的内容与最新修改一致。

### 6. 删除任务

**操作步骤**

1. 输入：`删除“面试作业”任务`
2. 再输入：`列出我的任务`

**预期结果**

- 聊天区提示任务已删除。
- 左侧任务列表不再展示该任务。

### 7. 普通联网搜索

**操作步骤**

1. 输入：`请联网搜索一下 Cloudflare Worker 最新 AI 能力`
2. 观察回复内容。

**预期结果**

- 系统触发外部搜索链路。
- 回复中应体现搜索结果整理，而不是单纯基于模型记忆泛答。
- 内容应偏向实时信息整合。

### 8. 深度研究与子任务规划

**操作步骤**

1. 输入：`帮我调研 Cloudflare Worker 上做 RAG 的轻量实现方案`
2. 观察聊天区的即时回复。
3. 观察研究进度卡片中的状态、进度条、阶段切换。
4. 等待研究完成。
5. 完成后点击 `复制报告` 或 `导出 Markdown`。

**预期结果**

- 系统不会直接单轮回答，而是进入研究模式。
- 聊天区会说明研究将被拆解执行。
- 研究卡片会展示步骤推进和阶段更新。
- 最终输出结构化研究报告。
- 报告中最好包含执行摘要、关键发现、推荐方案、参考来源等结构化章节。

### 9. 上传文件

**操作步骤**

1. 在左侧文件工作区选择 `resume.md`。
2. 点击上传。
3. 再上传 `notes.txt`。
4. 观察上传状态提示。

**预期结果**

- 上传成功后会显示“解析与向量化完成”。
- 状态文案中应包含切分片段数量和向量数量。
- 文件列表中会新增对应文件。

### 10. 文件问答与 RAG

**操作步骤**

1. 在文件列表中勾选 `resume.md`。
2. 输入：`总结这个文档的核心内容`
3. 再输入：`这个文档里和 Agent、RAG 相关的经历有哪些？`

**预期结果**

- 回复应明显基于上传文档内容。
- 不是泛泛而谈，而是能提炼出文件中真实存在的要点。
- 页面会提示当前聊天带上了选中文件上下文。

### 11. 多文件联合问答

**操作步骤**

1. 同时勾选 `resume.md` 和 `notes.txt`。
2. 输入：`结合这两个文件，帮我整理一份面试讲稿提纲`

**预期结果**

- 回答会综合多个文件内容。
- 能体现系统支持多文件 RAG 上下文联合检索。

### 12. 查看文件详情

**操作步骤**

1. 在文件列表中点击某个文件右侧的 `查看详情`。
2. 观察详情卡片。

**预期结果**

- 页面展示文件名、类型、大小、向量数、创建时间。
- 能看到摘要或预览内容。

### 13. 重命名文件

**操作步骤**

1. 点击某个文件右侧的 `重命名`。
2. 将 `resume.md` 改为 `resume-final.md`。
3. 注意保持扩展名不变。
4. 重命名后重新查看文件列表，并继续提问：`总结这个文档`

**预期结果**

- 文件名更新成功。
- 文件仍然可以继续用于问答。
- 说明文件元数据和向量引用已同步更新。

### 14. 删除文件

**操作步骤**

1. 点击文件右侧 `删除文件`。
2. 删除 `resume-final.md`。
3. 再次提问：`继续总结这个文档`
4. 或刷新文件列表。

**预期结果**

- 文件从文件工作区消失。
- 删除后不应再召回已删除文件的内容。
- 这说明文件元数据、原始文件和向量检索结果已联动清理。

### 15. 新会话测试

**操作步骤**

1. 点击顶部 `新会话`。
2. 观察聊天区是否清空。
3. 再输入：`列出我的任务`

**预期结果**

- 聊天历史从当前页面清空，进入新会话。
- 用户身份、文件、任务等持久化数据仍然保留。
- 可以向面试官说明：会话上下文和业务数据持久化是分离的。

### 16. 清空数据测试

**操作步骤**

1. 点击顶部 `清空数据`。
2. 在弹窗中确认。
3. 等待操作完成。
4. 再查看任务列表、文件列表与研究区。

**预期结果**

- D1、R2、Qdrant 对应的数据被清空。
- 页面回到干净状态。
- 适合多轮演示之间重置环境。

## 四、推荐演示顺序

建议将整个演示控制在 5 到 8 分钟。

### 简版演示顺序

1. 打开首页，直接说“帮我创建一个任务”，展示系统先追问资料。
2. 输入姓名和邮箱，再把 AI 改名为“阿塔”。
3. 创建一个与作业强相关的任务。
4. 上传 `resume.md`，勾选文件后问“总结这个文档”。
5. 展示文件重命名。
6. 发起一个研究问题。
7. 展示研究进度与最终报告。
8. 删除文件，证明删除后不会继续召回。

## 五、建议向面试官重点强调的能力点

- 系统是对话式任务管理，不依赖传统表单。
- 用户身份信息会先补全，再进入正常任务执行。
- 文件上传后不仅存储，还完成了解析、切块、向量化与检索增强回答。
- 复杂研究问题不是单轮回答，而是拆成多个子问题逐步推进。
- 持久化层覆盖用户资料、任务、会话、文件与向量记忆。
- 文件删除后的“不可再召回”是一个很好的工程一致性展示点。
