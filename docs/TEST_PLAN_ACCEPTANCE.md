# 智能对话式任务管理助手测试方案与验收指标

## 1. 文档目的

本文档用于支撑本项目的作业交付与最终验收，目标是回答两个问题：

- 这个系统应该如何测试，才能证明核心功能真实可用。
- 项目达到什么标准，才算满足需求文档并可交付演示。

本文档覆盖以下范围：

- 对话式用户管理
- 自然语言任务管理
- 实时搜索与深度研究
- 文件上传、解析、向量化与 RAG
- 数据持久化与多租户隔离
- Cloudflare Worker 部署后的可访问性与稳定性

## 2. 测试目标

本项目测试不只验证“接口能返回”，而是验证以下四类目标：

1. 功能闭环成立。用户可以只通过对话完成信息补全、任务管理、研究、文件问答和文件管理。
2. Agent 能力成立。系统具备工具调用、记忆召回、子任务规划与结果整合能力。
3. 数据链路成立。D1、R2、Qdrant 三类存储在新增、查询、更新、删除场景下保持一致。
4. 部署形态成立。系统在 Cloudflare Worker 上可公开访问，并完成一次真实在线演示。

## 3. 测试范围与分层

### 3.1 测试范围

纳入本次测试的模块：

- Web UI 与对话入口
- 会话与用户资料补全逻辑
- 助手昵称设置与持久化
- 任务意图识别与任务 CRUD
- 搜索服务与网页抓取
- 深度研究规划、执行、轮询与报告生成
- 文件上传、解析、切块、向量化、检索、重命名、删除
- 对话记忆与上下文压缩
- D1 / R2 / Qdrant 适配层

本次不作为硬性阻塞项，但可作为加分验收的模块：

- 图片 OCR、多模态输入
- ToT 风格研究结论对比
- 前端文件工作区体验优化

### 3.2 测试分层

建议按四层执行测试：

1. 单元测试
   验证意图解析、文本分块、上下文管理、模型适配、HTML 清洗等纯逻辑模块。
2. 服务集成测试
   验证 Agent、任务服务、研究服务、文件服务、RAG 服务等模块间协作。
3. 端到端验收测试
   从浏览器入口出发，验证真实用户路径是否闭环。
4. 部署后冒烟测试
   针对 Cloudflare Worker 公网地址做上线后核验，确认环境变量、外部 API、数据库绑定生效。

## 4. 测试环境方案

建议至少准备 3 套环境。

| 环境 | 用途 | 依赖 |
| --- | --- | --- |
| 本地开发环境 | 快速开发、自测、单元测试 | SQLite / 本地 R2 适配 / 本地向量存储 |
| 预发布环境 | 验证 Worker 绑定与外部服务联通 | D1 / R2 / Qdrant / Serper / LLM API |
| 生产演示环境 | 面试演示与最终验收 | Worker 公网域名 |

建议准备两类测试数据：

- 正常数据：姓名、邮箱、中文任务描述、PDF/TXT/DOCX/图片文件、研究型问题。
- 异常数据：空消息、超长消息、重复上传、非法文件类型、超大文件、缺失外部 API 密钥。

## 5. 自动化测试方案

### 5.1 自动化测试目标

自动化测试承担两个职责：

- 对核心能力做回归保护，避免重构后破坏闭环。
- 为面试验收提供可重复执行的证据。

### 5.2 建议执行命令

若本地已安装开发依赖，执行：

```bash
uv run pytest -q
```

若使用系统 Python 环境，则执行：

```bash
python3 -m pytest
```

为了更接近最终作业要求，建议补充以下自动化测试：

   覆盖 `/chat`、`/tasks`、`/files`、`/research` 的状态码、参数校验和错误返回。

   用脚本自动访问公开 URL，验证首页加载、接口联通和关键文本存在。

## 6. 手工测试方案

手工测试主要覆盖真实用户视角下的完整闭环，因为这类体验问题很难仅靠单元测试发现。

### 6.1 基础对话与用户管理

**目标**

- 未提供姓名和邮箱时，助手先追问资料，不直接执行任务。
- 用户补全资料后，后续对话正确称呼用户。
- 用户可以给助手改名，且刷新页面后仍生效。

**关键用例**

| 用例编号 | 场景 | 预期结果 |
| --- | --- | --- |
| U-01 | 首次访问直接说“帮我建一个任务” | 助手先追问姓名和邮箱，任务不落库 |
| U-02 | 用户回复“我叫小李，邮箱是 xiaoli@example.com” | 资料写入 D1，助手确认已记住 |
| U-03 | 再次发送普通消息 | 助手能正确称呼“小李” |
| U-04 | 用户说“以后叫你阿塔” | 助手昵称即时更新并持久化 |
| U-05 | 刷新页面重新进入会话 | 用户资料与助手昵称仍然存在 |

### 6.2 任务管理

**目标**

- 用户可以只通过自然语言完成任务的增删改查。
- 任务标题、需求、截止时间、优先级能被正确提取并持久化。

**关键用例**

| 用例编号 | 场景 | 预期结果 |
| --- | --- | --- |
| T-01 | “创建一个简历优化任务，下周五前完成，高优先级” | 任务创建成功，字段解析正确 |
| T-02 | “给这个任务补充要求：突出 Agent、RAG、Cloudflare Worker 项目经验” | `details` 被更新 |
| T-03 | “列出我所有未完成任务” | 返回任务列表且只显示当前用户任务 |
| T-04 | “把简历优化改成中优先级” | 优先级更新成功 |
| T-05 | “删除简历优化任务” | 任务被删除，后续查询不再出现 |

### 6.3 外部搜索与深度研究

**目标**

- 普通实时问题能触发搜索并返回外部信息。
- 复杂问题能触发规划、拆题、搜索、整合和报告输出。

**关键用例**

| 用例编号 | 场景 | 预期结果 |
| --- | --- | --- |
| R-01 | “Cloudflare Workers 最近支持哪些 AI 相关能力？” | 返回搜索整合结果，并带来源 |
| R-02 | “帮我调研 Cloudflare Worker 上做 RAG 的轻量实现方案” | 创建 research job，前端展示进度 |
| R-03 | 研究执行中刷新页面 | 仍可轮询到相同任务状态 |
| R-04 | 研究完成后查看结果 | 输出结构化 Markdown 报告 |
| R-05 | 报告中检查内容 | 至少包含执行摘要、关键发现、方案对比、推荐方案、参考来源 |

### 6.4 文件处理与 RAG

**目标**

- 文件能上传、解析、切块、向量化并进入检索链路。
- 用户可在工作区管理文件。
- 问答时能从文件中召回相关内容。

**关键用例**

| 用例编号 | 场景 | 预期结果 |
| --- | --- | --- |
| F-01 | 上传 `txt/md/pdf/docx` 文件 | 文件列表出现新文件，状态成功 |
| F-02 | 上传后立即提问“总结这个文档” | 回答基于文档内容，不是泛泛而谈 |
| F-03 | 上传多个文件后指定其中一个提问 | 检索范围受 `file_id` 约束 |
| F-04 | 在工作区重命名文件 | 文件名更新，向量 metadata 同步 |
| F-05 | 删除文件 | 文件元数据、原始文件、向量一并删除 |
| F-06 | 再次询问已删除文件内容 | 不应再召回已删除内容 |

### 6.5 多模态加分项

若实现图片 OCR 或其他多模态能力，建议补充：

| 用例编号 | 场景 | 预期结果 |
| --- | --- | --- |
| M-01 | 上传图片并提问图片中有什么文字 | OCR 提取成功，答案包含识别结果 |
| M-02 | 图片 OCR 结果进入 RAG | 后续对话可召回该文本 |

## 7. 非功能测试方案

### 7.1 可用性

- 首页在桌面端和移动端均可打开。
- 首屏加载后用户知道如何开始对话、上传文件、查看任务和研究结果。
- 错误提示清晰，例如缺少邮箱、上传失败、解析失败、研究中等状态都有明确反馈。

### 7.2 性能

建议记录以下指标：

| 指标 | 建议目标 |
| --- | --- |
| 首页首屏可交互时间 | < 3 秒 |
| 普通聊天接口平均响应时间 | < 8 秒 |
| 任务 CRUD 接口平均响应时间 | < 2 秒 |
| 文件上传后开始进入处理中状态 | < 3 秒 |
| 深度研究任务提交响应时间 | < 3 秒 |

说明：

- 深度研究总完成时长不设严格硬门槛，因为它取决于外部搜索与模型 API。
- 对研究任务，重点验收“可提交、可追踪、可完成、可查看结果”，而不是追求固定秒级完成。

### 7.3 稳定性与容错

建议重点验证以下异常场景：

- Serper API 不可用时，系统是否能给出降级提示。
- 模型 API 超时或失败时，界面是否有明确错误反馈。
- Qdrant 写入失败时，文件是否标记为失败而不是假成功。
- D1 暂时不可写时，任务/资料是否避免出现“UI 成功但数据未落库”的假象。
- 重复刷新研究页面时，是否不会重复创建同一个研究任务。

### 7.4 安全与隔离

至少验证以下内容：

- 不同 `client_id` 之间无法互相查看任务、文件、对话、研究记录。
- Qdrant 检索必须带 `user_id` 过滤。
- 文件删除后不能再通过旧 `file_id` 命中向量内容。
- 前端不暴露服务端密钥。

## 8. 验收指标

### 8.1 功能验收指标

| 一级能力 | 验收指标 | 通过标准 |
| --- | --- | --- |
| 对话与用户管理 | 首次追问资料、资料补全、正确称呼、昵称可修改 | 5 个核心用例全部通过 |
| 任务管理 | 增删改查、需求提取、优先级/时间解析、持久化 | 5 个核心用例全部通过 |
| 搜索与研究 | 普通搜索可用，深度研究可拆题、可跟踪、可生成报告 | 5 个核心用例全部通过 |
| 文件与 RAG | 上传、解析、检索、引用、重命名、删除 | 6 个核心用例全部通过 |
| 数据持久化 | 刷新页面或重新进入会话后，资料/任务/文件仍存在 | 至少 3 类数据均验证通过 |

### 8.2 自动化验收指标

建议将自动化测试作为交付门禁：

- P0 自动化测试通过率：100%
- 全量自动化测试通过率：100%
- 新增功能必须补至少 1 个对应测试
- 研究、文件、任务三条主链路至少各有 1 个集成测试

说明：

- P0 指影响主流程闭环的缺陷，例如无法聊天、无法建任务、无法上传文件、研究任务无法完成、数据不持久。
- 若存在 P0 缺陷，则本次验收直接不通过。

### 8.3 缺陷分级与验收门禁

| 缺陷等级 | 定义 | 验收要求 |
| --- | --- | --- |
| P0 | 主流程不可用、数据串号、严重数据丢失、部署不可访问 | 0 个 |
| P1 | 主流程可走通但结果明显错误，或高概率失败 | 0 个 |
| P2 | 局部功能异常，有替代路径 | <= 3 个，且不影响演示主线 |
| P3 | UI 文案、样式、低频边界问题 | 可记录待优化，不阻塞提交 |

### 8.4 研究能力专项验收指标

这是本作业中最能体现 Agent 设计能力的部分，建议单独验收：

- 系统能够识别研究型问题并切换到 research mode。
- 研究计划至少拆分为 3 个子问题。
- 报告至少包含 5 个结构化章节。
- 报告包含来源链接或参考来源列表。
- 用户能看到中间进度，而不是长时间无反馈。

### 8.5 RAG 能力专项验收指标

- 文件上传成功后可见 `chunk_count` 或等价处理结果。
- 检索结果必须受 `user_id` 过滤。
- 指定文件问答时必须受 `file_id` 过滤。
- 回答内容与文件事实一致，不得明显脱离文档内容。
- 删除文件后，该文件相关向量不能继续被召回。

## 9. 最终验收流程

建议按以下顺序进行最终验收：

1. 验证公开网址可访问
2. 演示首次对话，补全姓名和邮箱
3. 修改助手昵称并刷新页面确认持久化
4. 完成一次任务创建、修改、查询、删除
5. 完成一次普通搜索
6. 完成一次深度研究，并展示研究进度和最终报告
7. 上传一个文件并进行 RAG 问答
8. 重命名并删除该文件，确认检索结果同步失效
9. 展示自动化测试结果与关键实现说明文档

## 10. 简单版端到端测试

这一部分适合面试现场演示，目标不是覆盖所有边界，而是用 2 到 3 条最短路径证明系统已经具备“对话 + 文件 + RAG + 研究”的完整闭环。

### 10.1 E2E-1：多文档上传与知识总结

**测试目标**

- 验证用户能上传多个文档。
- 验证系统能基于上传文档做跨文档知识总结。
- 验证回答内容来自文档，而不是泛泛生成。

**测试准备**

- 上传 2 到 3 个主题相关文档，例如：
  - `Cloudflare_Workers_Intro.pdf`
  - `RAG_Architecture_Notes.md`
  - `Agent_Design_Checklist.docx`

**测试步骤**

1. 打开公开网页，进入对话页面。
2. 若系统首次访问要求补全资料，则先输入姓名和邮箱。
3. 依次上传上述 2 到 3 个文档。
4. 等待文件处理完成，确认文件列表中能看到全部文件。
5. 在对话框输入：
   `请基于我上传的这些资料，总结这个项目涉及的核心知识点，并按“Agent、RAG、Cloudflare Worker、工程实现”四个部分整理。`

**预期结果**

- 系统能识别多个已上传文件。
- 返回结构化总结，而不是一句话泛答。
- 总结内容能覆盖多个文档中的信息点。
- 若系统支持引用或来源标识，回答中应体现来源痕迹。

### 10.2 E2E-2：基于已有文档继续做外部调研

**测试目标**

- 验证系统能把“本地知识库 + 外部搜索”串起来。
- 验证研究模式可以围绕已有材料继续扩展，而不是只复述文档。

**测试步骤**

1. 在完成 E2E-1 后继续提问：
   `基于我上传的文档，再帮我调研一下业界在 Cloudflare Worker 上做 RAG 的最佳实践，并告诉我当前方案还缺什么。`
2. 观察系统是否进入研究模式或显示研究进度。
3. 等待研究完成并查看最终报告。

**预期结果**

- 系统启动 research job 或等价深度研究流程。
- 输出内容同时包含：
  - 对现有上传资料的归纳
  - 对外部资料的补充调研
  - 对当前方案不足项的分析
- 最终结果至少包含“现状总结、外部最佳实践、差距分析、建议动作”四部分。

### 10.3 E2E-3：从总结结果继续沉淀任务

**测试目标**

- 验证系统能把知识总结和研究结论进一步转化为可执行任务。
- 验证任务管理与研究能力是连通的，而不是彼此孤立。

**测试步骤**

1. 在 E2E-2 研究完成后继续输入：
   `请根据刚才的总结和调研结果，帮我生成一个任务清单，拆成“本周必须完成”和“后续优化”两组。`
2. 再输入：
   `把“补齐 Qdrant 真实集成测试”设为高优先级，本周完成。`
3. 最后输入：
   `列出我当前的全部任务。`

**预期结果**

- 系统能把总结和调研结论转换成任务项。
- 新建任务能够写入任务列表。
- 任务优先级和时间要求可以继续通过自然语言修改。
- 最终查询结果中能看到刚刚创建和更新过的任务。

### 10.4 简化版验收标准

如果时间有限，建议面试现场至少完成以下 3 个动作：

1. 上传 2 到 3 个文档并成功总结知识。
2. 基于上传文档继续做一次外部调研。
3. 将调研结论转成任务清单并成功查询出来。

只要这 3 步能顺畅走通，就已经足以说明系统具备以下核心能力：

- 文件上传与解析
- RAG 检索与知识总结
- 外部搜索与深度研究
- 自然语言任务管理
- 多轮对话上下文衔接
